> 나머지는 전부 밥이다.

## 새로운 아키텍처가 매년 나오는 이유

Clean Architecture, Hexagonal Architecture, DDD, Onion Architecture...

매년 새로운 이름이 나온다. 컨퍼런스에서 발표되고, 책이 나오고, 블로그에 글이 올라온다. 마치 혁신적인 것처럼. 그런데 한번 까보자.

```
Clean Architecture   → 다형성으로 의존성 방향을 뒤집고, 캡슐화로 레이어를 분리
Hexagonal Architecture → 추상화로 포트를 정의하고, 다형성으로 어댑터를 교체
Onion Architecture   → 캡슐화로 도메인을 보호하고, 다형성으로 외부를 분리
DDD                  → 캡슐화로 도메인 모델을 격리하고, 추상화로 경계를 정의
```

전부 OOP다. 캡슐화, 추상화, 다형성. 이름만 다르고 본질은 같다.

1960년대부터 있던 OOP의 핵심 개념을, 누군가 다이어그램을 바꿔 그리고, 새 이름을 붙이고, 책을 내고, 컨설팅을 판다. 그게 전부다.

## 왜 아무도 이걸 말하지 않는가

돈이 안 되니까.

| 활동 | OOP 원리 | 새 버즈워드 |
|------|---------|-----------|
| 책 출판 | 이미 나올 거 다 나옴 | 매년 새 책 가능 |
| 컨퍼런스 발표 | "OOP 제대로 하세요" → 한 마디로 끝 | "XX 아키텍처 소개!" → 세션 가능 |
| 컨설팅 | "원리 지키세요" → 계약 불가 | "우리 방법론 도입하세요" → 계약 |
| 커리어 | 이력서에 안 씀 | "DDD 전문가" → 타이틀 |

본질은 마케팅이 안 된다. "새로운 것"이 팔리는 구조에서, OOP 같은 본질은 화제가 될 수 없다. 알고 있는 사람들은 조용히 적용하고, 모르는 사람들은 새 버즈워드를 쫓는다.

## 까보면 다 똑같다

실제로 해외 시니어 개발자들 사이에서는 이런 말이 은근히 나온다.

- "Clean Architecture is just SOLID with extra diagrams"
- "DDD is overengineered OOP"
- "Hexagonal is just dependency inversion with ports"

근데 이게 화제가 되지 않는다.

```
이미 아는 사람: "당연한 소리를 뭘 굳이"  → 글 안 씀
모르는 사람:    "무슨 소리야"             → 관심 없음
업계:          "새 방법론 팔아야 하는데"   → 묻어버림
```

## OOP가 전부인 이유

OOP의 핵심은 4가지다.

- **캡슐화**: 상태와 행위를 묶고, 내부를 숨긴다
- **추상화**: 복잡한 현실을 필요한 부분만 모델링한다
- **상속**: 기존 객체의 구조와 행위를 물려받는다
- **다형성**: 같은 인터페이스로 다른 동작을 실행한다

이 4가지를 제대로 이해하면, 어떤 아키텍처가 나와도 "아, 다형성으로 의존성 뒤집은 거네" 하고 끝이다. 새로 공부할 게 없다.

그리고 OOP를 **잘** 하기 위한 설계 원칙이 SOLID다.

```
OOP  = 도구 (이런 게 있다)     → "할 수 있다"
SOLID = 원칙 (이렇게 써라)     → "이렇게 해야 한다"
```

OOP가 망치와 톱이라면, SOLID는 "망치는 이렇게 잡고, 톱은 이 각도로 밀어라"다. 도구를 알면 원칙은 자연스럽게 따라온다. 도구도 모르는데 원칙만 외우면 아무 소용 없다.

```
원리를 아는 사람:    새 아키텍처 나옴 → "다형성이네" → 끝
원리를 모르는 사람:  새 아키텍처 나옴 → 책 삼 → 강의 들음 → 또 나옴 → 반복
```

## "OOP는 객체지향 전용 아닌가?"

아니다. 이름이 그럴 뿐이다.

OOP가 정리된 시대를 보자. 1960~80년대, 프로그래밍 세계는 **절차지향 vs 객체지향**이 싸우고 있었다. 함수형 프로그래밍은 이론으로는 존재했지만 주류가 아니었다. 껴들 자리가 없었다. 철저하게 소외됐다.

그 결과, 프로그래밍의 보편적인 원리가 **"객체지향"이라는 이름 안에 갇혀버렸다.**

```
"내부를 숨겨라"              → 어디서든 필요한 원리 → OOP에선 "캡슐화"
"구체가 아닌 계약에 의존해라"  → 어디서든 필요한 원리 → OOP에선 "다형성"
"복잡한 건 단순하게 노출해라"  → 어디서든 필요한 원리 → OOP에선 "추상화"
```

이 원리들은 객체지향에서만 작동하는 게 아니다.

```
JavaScript:  클로저로 내부를 숨긴다       → 캡슐화와 같은 원리
Go:          소문자로 비공개 처리한다      → 캡슐화와 같은 원리
함수형:       고차 함수로 동작을 교체한다   → 다형성과 같은 원리
Rust:        트레이트로 계약을 정의한다    → 추상화와 같은 원리
```

도구(문법)는 다르지만 해결하려는 문제는 같다.

**원리가 먼저고, OOP는 그 원리에 이름을 붙인 것이다.** 근데 그 이름이 "객체지향 프로그래밍"이다 보니, 마치 객체지향에서만 통하는 것처럼 보이는 것이다.

그래서 이 글에서 "OOP를 공부해라"라고 말하는 건, "class 키워드를 써라"가 아니다. **OOP라는 이름으로 정리된 보편적 원리를 이해해라**는 뜻이다. 그 원리를 알면 어떤 패러다임에서든 적용할 수 있다.

## 10년간 배운 게 "고작" OOP?

겉보기에는 초라해 보인다.

```
대부분의 10년차 이력서:
  Flutter, React, Spring, Django, Kubernetes, Docker,
  MSA, DDD, TDD, CI/CD, AWS, GCP...
  → 2페이지
  → 근데 "왜 이걸 쓰는지" 설명 못 함

원리를 아는 사람 이력서:
  OOP + SOLID
  → 한 줄
  → 근데 뭐가 나와도 본질을 꿰뚫음
```

기술 이름 10개를 외운 사람보다, 원리 하나를 제대로 이해한 사람이 강하다. 기술은 바뀌지만 원리는 바뀌지 않기 때문이다.

## 6개월 코스의 환상

6개월 코딩 부트캠프가 쏟아지는 이유도 같은 구조다.

```
기업: 싼 인력 필요 → 6개월이면 충분
정부: 취업률 통계 필요 → 수료 = 성과
학원: 수강생 = 매출 → 많을수록 좋음
수강생: 취업 필요 → 빨리 배우고 싶음
```

모두의 이해관계가 맞아떨어진다. 누구도 "원리"를 가르칠 이유가 없다. 원리는 시간이 걸리고, 시간은 돈이고, 돈은 빨리 벌어야 하니까.

하지만 6개월 만에 배운 기술은 2년 뒤에 바뀐다. 원리를 모르면 또 6개월을 써야 한다. 그 반복을 10년 하면, "경력 10년차인데 매번 처음부터 배우는 사람"이 된다.

## 결론

새로운 아키텍처가 나오면 흥분하기 전에 까봐라.

```
1. 메커니즘을 본다: "이게 실제로 뭘 하는 거지?"
2. 원리로 환원한다: "이건 OOP의 어떤 개념이지?"
3. 차이를 확인한다: "기존과 본질적으로 다른 게 있나?"
```

대부분은 3번에서 끝난다. 본질적으로 다른 게 없다.

**원리를 공부해라. 나머지는 전부 밥이다.**
