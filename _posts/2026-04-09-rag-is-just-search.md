> 까보면 그냥 검색이다.

## RAG가 뭔데

RAG(Retrieval-Augmented Generation). 이름부터 거창하다. "검색 증강 생성". AI 기술처럼 들린다.

한번 까보자.

```
RAG의 동작 방식:
1. 문서를 벡터로 변환해서 저장한다
2. 질문이 들어오면 벡터 유사도로 관련 문서를 찾는다
3. 찾은 문서를 LLM 프롬프트에 붙여서 답변한다
```

이게 전부다.

## 이거 검색 아닌가?

맞다. 그냥 검색이다.

```
전통적 검색:  키워드 → 인덱스 조회 → 결과 반환
RAG:         벡터 → 유사도 조회 → 결과를 프롬프트에 붙임
```

차이점은 "키워드 매칭" 대신 "벡터 유사도"를 쓴다는 것. 그리고 결과를 사람에게 직접 보여주는 대신 LLM에게 넘긴다는 것. 그게 전부다. AI 기술이 아니라 **검색 기법**이다.

## 구조화된 데이터에서 RAG는 차선책이다

데이터가 구조화되어 있으면 RAG가 필요 없다. 아니, 쓰면 안 된다.

```
질문: "3월 매출이 얼마야?"

RAG 방식:
  1. "3월 매출"을 벡터로 변환
  2. 유사한 문서 검색 → "3월 보고서.pdf"의 일부 발췌
  3. LLM에게 "이 문서에서 매출을 찾아줘"
  4. 할루시네이션 위험 + 정확도 불확실

SQL 방식:
  SELECT SUM(amount) FROM sales WHERE month = 3;
  → 정확한 숫자 → 끝
```

어느 쪽이 더 정확하고 빠른가? 비교할 가치도 없다.

## RAG가 빛을 발하는 유일한 상황

**구조화할 수 없는 비정형 데이터**를 다룰 때.

```
RAG가 적합한 경우:
  - 수천 개의 PDF 매뉴얼에서 정보 찾기
  - 레거시 문서 더미에서 답변 생성
  - 자유형 텍스트가 주 데이터인 경우

RAG가 부적합한 경우:
  - DB에 정리된 데이터 조회
  - 정확한 숫자/통계 질의
  - 구조화 가능한 모든 데이터
```

즉, RAG는 **"데이터를 정리 못 했을 때 쓰는 차선책"**이다. 최선이 아니다.

## 그런데 왜 모두가 RAG를 쓰는가

```
1. AI 시대에 "AI스러운" 기술을 쓰고 싶다
   SQL은 AI 같지 않다. 벡터 검색은 AI 같다.

2. 구조화하기 귀찮다
   데이터 정리하는 데 몇 달 걸린다. RAG는 문서 던지면 된다.

3. 기술 스택이 화려해 보인다
   "벡터 DB + 임베딩 + RAG 파이프라인" → 있어 보인다.
   "SQL 쿼리 몇 개" → 없어 보인다.

4. 돈이 된다
   RAG 관련 SaaS, 프레임워크, 컨설팅 → 시장이 생긴다.
   SQL → 이미 있다. 팔 게 없다.
```

역시나 돈의 문제다.

## Tool Use가 더 낫다

LLM에게 도구(Tool)를 쥐어주면, RAG 없이도 정확한 답을 할 수 있다.

```
RAG 방식:
  질문 → 벡터 검색 → 비슷한 문서 발췌 → LLM이 추론 → 답변
  (부정확할 수 있음, 할루시네이션 가능)

Tool Use 방식:
  질문 → LLM이 적절한 도구 선택 → SQL/API 호출 → 정확한 데이터 → 답변
  (정확함, 검증 가능)
```

실제 예시:

```
사용자: "지난달 신규 가입자 수 알려줘"

RAG: 가입 관련 문서 검색 → "약 1,200명 정도입니다" (추정)
Tool: SELECT COUNT(*) FROM users WHERE created_at >= '2026-03-01' → 1,247명 (사실)
```

데이터가 구조화되어 있다면, LLM에게 도구를 주는 것이 RAG보다 항상 낫다.

## 근데 왜 다들 Tool Use 안 하고 RAG 하는가

어렵기 때문이다.

```
RAG 구현:
  1. 문서를 벡터 DB에 넣는다
  2. 검색한다
  3. 프롬프트에 붙인다
  → 프레임워크가 다 해줌 (LangChain, LlamaIndex 등)

Tool Use 구현:
  1. 데이터를 구조화한다
  2. API/쿼리 인터페이스를 설계한다
  3. LLM이 올바른 도구를 선택하도록 설계한다
  4. 에러 처리, 권한 관리, 보안까지 고려한다
  → 직접 설계해야 함
```

RAG는 쉽고, Tool Use는 어렵다. 쉬운 길이 항상 먼저 퍼진다.

## 까보기 체크리스트

새 기술이 나왔을 때:

```
1. 메커니즘을 본다
   "이게 실제로 뭘 하는 거지?"
   → RAG: 벡터 검색이다

2. 기존 방법과 비교한다
   "이전에는 이 문제를 어떻게 풀었지?"
   → SQL, API, 전문 검색(Elasticsearch)

3. 본질적 차이를 확인한다
   "기존 방법으로 못 하는 게 있나?"
   → 비정형 데이터 검색에서만 유리

4. 결론을 내린다
   → RAG: 비정형 데이터의 차선책. 구조화 가능하면 쓸 이유 없음.
```

## 결론

RAG는 AI 기술이 아니다. 검색이다.

데이터가 구조화되어 있으면 SQL이 더 정확하고, Tool Use가 더 강력하다. RAG는 **"데이터를 정리 못 했을 때의 차선책"**이지, 만능 해결사가 아니다.

까봐라. 이름에 속지 마라.

**구조화할 수 있으면 구조화해라. RAG는 그다음이다.**
